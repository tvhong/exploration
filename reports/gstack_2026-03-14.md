# gstack

> A collection of opinionated Claude Code workflow skills — plus a persistent headless browser daemon — designed to give the AI agent specialized cognitive modes and vision of the running app.

## What is it?

gstack was built by Garry Tan (President & CEO of Y Combinator) to solve a specific pain with AI coding assistants: they default to one generic mode. Planning, review, shipping, and QA all require different kinds of thinking, but a vanilla Claude Code session blends them into a mediocre average. The agent also has no visibility into the actual running application — it cannot see the UI, verify that a page loads, or catch a broken layout.

gstack answers both problems:

1. **Workflow skills** — eight slash commands, each instructing Claude to adopt a specific cognitive posture: founder taste (`/plan-ceo-review`), engineering rigor (`/plan-eng-review`), paranoid review (`/review`), automated release (`/ship`), QA testing (`/qa`), browser session management (`/setup-browser-cookies`), and retrospective (`/retro`).
2. **Persistent browser** — `/browse` and `/qa` talk to a compiled Chromium daemon. The agent can navigate pages, fill forms, take screenshots, read console errors, and check layouts — in roughly 100–200 ms per command after the first call.

The target user is already a heavy Claude Code user and wants consistent, high-rigor workflows instead of ad-hoc prompting. The README explicitly positions this as "an operating system for people who ship," not a beginner prompt pack.

## Ecosystem fit & innovations

**Where it sits:** gstack sits on top of Claude Code's slash-command / skills system. Each skill is a Markdown file with a YAML front-matter header (`name`, `version`, `description`, `allowed-tools`). Claude reads the file at invocation time and follows its instructions. gstack provides the files; it does not modify Claude Code itself.

**What makes it different from prompt packs:**

- **Compiled browser binary.** Rather than relying on MCP browser tools (which gstack explicitly prohibits — "never use `mcp__claude-in-chrome__*`"), it ships a ~58 MB Bun-compiled Playwright binary. This avoids the cold-start penalty (3–5 s per command for MCP tools) and keeps full state (cookies, localStorage, open tabs) between commands. The daemon model is genuinely novel for this class of tooling.

- **Ref system instead of DOM mutation.** Most browser automation for LLMs injects `data-ref` attributes into the DOM to give the agent stable handles. gstack uses Playwright's accessibility tree and `getByRole()` locators instead. This works on sites with strict CSP headers, React/Vue/Svelte hydration, and shadow DOM — all of which break DOM injection approaches.

- **SKILL.md template system.** Skill documentation is generated from `.tmpl` files that pull live metadata from `commands.ts` (the command registry) and `snapshot.ts` (the flag registry). If a command is added in code, it automatically appears in the docs. Stale docs are caught by CI (`--dry-run` + `git diff --exit-code`). This is a structural solution to documentation drift, applied to what are essentially AI prompt files.

- **Three-tier test pyramid for prompts.** Testing AI prompt quality as a first-class engineering concern is unusual. The three tiers are:
  - **Tier 1 — Static validation** (`skill-validation.test.ts`): Parses every `$B <cmd>` call in SKILL.md files and checks each against the live command registry; validates snapshot flag names; asserts no unresolved `{{placeholder}}` template variables. Free, <2 s. Runs in CI.
  - **Tier 2 — Agent SDK E2E** (`skill-e2e.test.ts`): Spins up a local test HTTP server, launches a real Claude session via the Anthropic Agent SDK, has the agent discover the browse binary and run commands against the server, and asserts no errors. ~$0.50/run. Opt-in via `SKILL_E2E=1`; auto-skipped when already running inside a Claude Code session (nested Agent SDK calls hang).
  - **Tier 3 — LLM-as-judge** (`skill-llm-eval.test.ts`): Sends sections of SKILL.md to Claude Sonnet and asks it to score clarity / completeness / actionability on a 1–5 scale, asserting all scores ≥ 4. Also runs a regression test comparing auto-generated docs against a hand-maintained baseline, asserting the generated version scores at least as high. ~$0.05–0.15/run.

- **Greptile integration.** [Greptile](https://greptile.com) is an AI-powered code review service that automatically posts comments on GitHub PRs as `greptile-apps[bot]`. `/review` and `/ship` fetch those comments via the GitHub API, classify each one (valid finding → act on it / already fixed → dismiss / false positive → dismiss + record), and maintain `~/.gstack/greptile-history.md` — a per-repo log so known false positives are auto-suppressed on future runs without re-review. The integration is additive; the workflow works fine without it.

- **Conductor integration.** [Conductor](https://conductor.build) is a Mac app that runs multiple Claude Code sessions simultaneously, each in its own **git worktree** (= isolated checkout with its own working directory and filesystem). gstack is designed to run ten such parallel sessions. Each workspace gets its own `.gstack/browse.json` state file and thus its own browser daemon with a randomly selected port (10,000–60,000) — no shared state, no port collisions, no configuration needed. The `bin/dev-setup` script handles the worktree case explicitly: it detects if it's running in a non-main worktree and copies `.env` from the main worktree so API keys carry over automatically. Conductor does not appear to have a public GitHub repository; it is a commercial product.

## Main components

| Component | What it does | Location |
|-----------|--------------|----------|
| **Workflow skills** | Markdown prompt files that instruct Claude to adopt specific modes | `plan-ceo-review/`, `plan-eng-review/`, `review/`, `ship/`, `retro/`, `qa/` |
| **browse CLI** | Thin compiled binary; reads `.gstack/browse.json`, starts/finds the server, sends HTTP POST | `browse/src/cli.ts` → `browse/dist/browse` |
| **browse server** | Long-lived `Bun.serve` HTTP daemon on localhost; owns the Playwright context | `browse/src/server.ts` |
| **BrowserManager** | Wraps a Playwright `Browser` + `Page`; holds the ref map (`@e1`, `@e2`, …); manages tabs | `browse/src/browser-manager.ts` |
| **Command registry** | Single source of truth for all 50+ browse commands (READ / WRITE / META sets) | `browse/src/commands.ts` |
| **Snapshot engine** | Produces the accessibility tree with `@e` refs; supports diff, annotated screenshot, cursor-interactive scan | `browse/src/snapshot.ts` |
| **Read/Write/Meta handlers** | Separate modules for the three command categories | `browse/src/read-commands.ts`, `write-commands.ts`, `meta-commands.ts` |
| **Cookie import** | Reads Chromium SQLite cookie DBs; decrypts via macOS Keychain (PBKDF2 + AES-128-CBC) | `browse/src/cookie-import-browser.ts` |
| **Cookie picker UI** | Local HTTP route serving a browser UI to select which domains to import | `browse/src/cookie-picker-routes.ts`, `cookie-picker-ui.ts` |
| **SKILL.md generator** | Build script that merges `.tmpl` prose with live command/flag metadata | `scripts/gen-skill-docs.ts` |
| **Skill validator** | Parses every `$B <cmd>` call in SKILL.md files; validates against command registry | `test/helpers/skill-parser.ts`, `test/skill-validation.test.ts` |
| **State file** | `.gstack/browse.json` — PID, port, bearer token, binary version hash | written by server at startup |
| **Log buffers** | Three `CircularBuffer` instances (console, network, dialog) flushed to disk every 1 s | `browse/src/buffers.ts` |
| **setup script** | Builds binary, symlinks skills into `~/.claude/skills/` | `setup` |

## How it works & data flow

### Skill invocation

```
User types /browse or /qa in Claude Code
         │
         ▼
Claude reads SKILL.md front-matter → restricts tools, gets instructions
         │
         ▼
Claude follows SKILL.md instructions → calls $B (browse binary) via Bash tool
```

### Browse command path (after first call)

```
Claude Code (Bash tool)
  │  $B snapshot -i
  ▼
browse CLI (compiled binary, ~1ms startup)
  1. Reads .gstack/browse.json → gets port + bearer token
  2. Checks binary version hash vs server's binaryVersion
     → mismatch: kills old server, starts new one
  3. POST http://127.0.0.1:<port>/command
     Authorization: Bearer <token>
     { "command": "snapshot", "args": ["-i"] }
  │
  ▼
browse server (Bun.serve on localhost)
  1. Validates bearer token (401 if wrong)
  2. Routes to handleMetaCommand("snapshot", ...)
  │
  ▼
Snapshot engine
  1. page.locator('body').ariaSnapshot() → ARIA tree as YAML-like text
  2. Two-pass parse: count role+name pairs, then assign @e1, @e2, ...
  3. Build Playwright Locator per ref (getByRole + nth for disambiguation)
  4. Store Map<ref, Locator> on BrowserManager
  5. Return formatted text with @e prefixes
  │
  ▼
server returns 200 plain text
  │
  ▼
CLI prints to stdout
  │
  ▼
Claude reads output, sees: "@e3 [button] "Submit""
  │
  ▼
Claude calls: $B click @e3
  │
  ▼
server resolves @e3 → Playwright Locator → locator.click()
```

### First call / server startup

```
CLI reads .gstack/browse.json
  → missing or PID dead → spawns server in background
  → polls /health up to 8 s
  → server starts Chromium (headless, via Playwright/CDP)
  → server writes .gstack/browse.json (atomic tmp+rename, mode 0o600)
  → CLI reads port+token, sends command
```

### Cookie import flow

```
User: /setup-browser-cookies github.com
         │
Claude runs: $B cookie-import-browser chrome --domain github.com
         │
         ▼
Server handler
  1. Copies Chrome's Cookies SQLite DB to /tmp (avoids lock conflicts)
  2. Queries cookies WHERE host LIKE '%github.com%'
  3. Reads encryption key from macOS Keychain (triggers dialog on first use)
  4. PBKDF2(key, salt, 1003 iterations) → AES-128-CBC decrypt each cookie value
  5. Calls playwright context.addCookies([...])
  6. Returns "Imported N cookies for github.com"
  → session now has real login cookies; /browse and /qa can test authenticated pages
```

### QA diff-aware mode

```
User: /qa
         │
Claude (following qa/SKILL.md)
  1. git diff main --name-only → list of changed files
  2. Infer affected routes from file paths (controllers, views, API routes)
  3. For each route: $B goto <url>, $B snapshot, $B screenshot, $B console
  4. Analyze screenshots (Claude reads the PNG via Read tool)
  5. Synthesize QA report with health score and issue list
```

### Ship workflow

```
User: /ship
         │
Claude (following ship/SKILL.md) — fully non-interactive except for:
  - MINOR/MAJOR version bump decisions
  - CRITICAL review findings
  - Greptile comments needing human judgment

Step 1: Pre-flight — check branch, git status
Step 2: git fetch + merge origin/main
Step 3: Run test suites in parallel
Step 3.25: Run LLM evals if prompt files changed
Step 3.5: Pre-landing review (reads review/checklist.md, checks diff)
Step 3.75: Fetch + triage Greptile PR comments
Step 4: Auto-bump VERSION (MICRO/PATCH auto; MINOR/MAJOR asks)
Step 5: Auto-generate CHANGELOG from diff + commit history
Step 6: Split into bisectable commits (infra → models → controllers → VERSION)
Step 7: git push -u origin <branch>
Step 8: gh pr create → output PR URL
```

## Additional notes

**Workflow skills vs. skills:** A *skill* in Claude Code is any slash command backed by a SKILL.md file. A *workflow skill* specifically encodes a complete multi-step process with ordering, conditionals, and human checkpoints — e.g., `/ship` has 8 ordered steps from preflight to PR creation. The distinction is between a tool/mode (like `/browse`) and a business process with state and branching logic.

**Browse binary:** The `browse/dist/browse` binary is compiled by gstack from `browse/src/cli.ts` using Bun. It is not part of Playwright — Playwright is a library that the gstack server uses internally. The binary is a thin HTTP client; all Playwright logic lives in the server (`browse/src/server.ts`).

**iOS equivalent:** There is no iOS equivalent to gstack's browser integration. The specific design — Playwright accessibility tree, `getByRole()` locators, persistent daemon — is web-specific. Closest analogues for iOS UI automation are XCUITest (native), Appium (WebDriver wrapper over XCUITest), and Detox (React Native). None offer the compiled-daemon-with-ref-system architecture gstack built.
